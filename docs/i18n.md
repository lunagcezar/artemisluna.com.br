# i18n — Client-Side Locale Detection

The site uses a lightweight client-side approach for locale-aware UI strings. No server-side routing or content restructuring is involved.

## How it works

1. **Translation map** — `src/i18n/labels.ts` exports a `ptLabels` record mapping collection names to their Portuguese translations. Any collection without an entry falls back to the English label derived from `formatSegment()`.

2. **Bilingual rendering** — Components render their default (English) label and embed the Portuguese translation in a `data-pt` attribute.

3. **Client-side swap** — An inline `<script>` in `AstroNavbar.astro` checks `navigator.language`. If it starts with `"pt"`, it iterates over all elements with `[data-pt]` and replaces their text content with the Portuguese value.

## Adding a new collection label

Edit `src/i18n/labels.ts`:

```ts
export const ptLabels: Record<string, string> = {
  art: "Arte",
  blog: "Blog", // only needed if the Portuguese spelling differs
};
```

The key must match the collection name (as defined in `src/content.config.ts`).

## Adding a new component with locale-aware labels

Following the same pattern used in `AstroNavbar.astro`:

```astro
---
import { ptLabels } from "../../../i18n/labels";

const label = "SomeLabel";
const pt = ptLabels["some-key"] ?? label;
---

<span data-en={label} data-pt={pt}>{label}</span>

<script>
  if (navigator.language.startsWith("pt")) {
    document.querySelectorAll("[data-pt]").forEach((el) => {
      el.textContent = el.getAttribute("data-pt");
    });
  }
</script>
```
