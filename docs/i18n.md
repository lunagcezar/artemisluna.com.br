# i18n — Client-Site Locale Detection

The site uses a lightweight client-side approach for locale-aware UI strings. No server-side routing or content restructuring is involved.

## How it works

1. **Translation map** — `src/i18n/labels.ts` exports `ptLabels` mapping collection names and `ptSegmentLabels` mapping taxonomy segments to their Portuguese translations. Any unmapped value falls back to the English label derived from `formatSegment()`.

2. **Bilingual rendering** — Components render their default (English) label and embed the Portuguese translation in a `data-pt` attribute.

3. **Client-side swap** — An inline `<script>` in `AstroNavbar.astro` checks the `locale` cookie first, then falls back to `navigator.language`. If Portuguese, it swaps all `[data-pt]` elements to their Portuguese value. The same script also applies the `theme` cookie or `prefers-color-scheme` for dark mode, preventing flash before React hydrates.

## Adding a new collection label

Edit `src/i18n/labels.ts`:

```ts
export const ptLabels: Record<string, string> = {
  art: "Arte",
};
```

The key must match the collection name (as defined in `src/content.config.ts`).

## Adding segment / tag translations

Segments (URL path parts like `gouache`, `oil-pastel`) and tag values are translated via `ptSegmentLabels` using dot-notation keys prefixed by collection name:

```ts
export const ptSegmentLabels: Record<string, string> = {
  "art.gouache": "Guache",
  "art.oil-pastel": "Pastel de óleo",
  "wiki.encryption": "Criptografia",
};
```

Create a `TranslateLabel` function via `createTranslateLabel(collection)` and pass it down to child components:

```astro
---
import { createTranslateLabel } from "@i18n/labels";
const translateLabel = createTranslateLabel("art");
---

<Component translateLabel={translateLabel} />
```

Child components receive `translateLabel: TranslateLabel` where `TranslateLabel = (segment: string) => { en: string; pt: string }` and render both languages as data attributes.
