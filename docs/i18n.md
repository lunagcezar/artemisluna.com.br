# i18n — Client-Site Locale Detection

The site uses a lightweight client-side approach for locale-aware UI strings. No server-side routing or content restructuring is involved. The system is designed to be **locale-agnostic** — every UI string, segment label, breadcrumb item, and date formatter iterates over `SUPPORTED_LOCALES` to produce labels for all configured locales. Adding a new locale to `SUPPORTED_LOCALES` automatically extends all i18n features.

## How it works

1. **Translation map** — `src/i18n/labels.ts` exports a single `translations` map keyed by locale (`en`, `pt`). Each locale maps keys (collection names, taxonomy segments, UI strings) to their translated value. Unmapped keys fall back to `formatSegment()` for segments or the raw key for UI strings.

2. **Locale-agnostic rendering** — Components render their default (English) label and embed all locale translations as a `data-locales` JSON attribute containing entries for every locale in `SUPPORTED_LOCALES`.

3. **Client-side swap** — A blocking `<script is:inline>` at the end of `<body>` in `MainLayout.astro` reads the `locale` cookie and swaps all `[data-locales]` elements to the correct language before paint.

## Page title translation

Directory index pages (e.g. `/wiki/linux/`, `/wiki/encryption/`) get translated page titles via `data-locales` on the `<h1>` element. The title is derived from the URL segment using `createTranslateLabel(collection)`, which produces labels for all supported locales. When a translation exists for a segment (e.g. `"wiki.encryption": "Criptografia"` in Portuguese), the `h1` switches on locale change.

The `<title>` meta tag in `<head>` also reflects the translated segment (e.g. `"Criptografia - Wiki"` when the Portuguese locale is active).

Detail pages keep their frontmatter `title` as-is — the original author-provided title is always shown regardless of UI locale.

## Adding a new locale

1. Add the locale code to `SUPPORTED_LOCALES`:
   ```ts
   export const SUPPORTED_LOCALES = ["en", "pt", "eo"] as const;
   ```
2. Add translations under the new key in `translations`:
   ```ts
   export const translations = {
     en: { light: "Light", dark: "Dark" },
     pt: {
       art: "Arte",
       "art.gouache": "Guache",
       light: "Claro",
       dark: "Escuro",
     },
     eo: { art: "Arto", "art.gouache": "Guŝo", light: "Lumo", dark: "Malhela" },
   };
   ```
3. Add the locale mapping to `src/lib/date.ts` `LOCALE_MAP` if the locale uses a different `Intl.DateTimeFormat` string:
   ```ts
   const LOCALE_MAP: Record<string, string> = {
     en: "en-US",
     pt: "pt-BR",
     eo: "eo",
   };
   ```
4. The switcher (`src/components/shared/_astro/AstroLocaleSwitcher.astro`) auto-generates its options from `SUPPORTED_LOCALES` — no manual UI changes needed.

`createTranslateLabel`, `t()`, and `formatDateForLocales()` automatically include all defined locales.

## Adding segment / tag translations

Segments use dot-notation keys prefixed by collection name:

```ts
pt: {
  "art.gouache": "Guache",
  "art.oil-pastel": "Pastel de óleo",
  "wiki.encryption": "Criptografia",
}
```

Create a `TranslateLabel` function via `createTranslateLabel(collection)` and pass it to child components:

```astro
---
import { createTranslateLabel } from "@i18n/labels";
const translateLabel = createTranslateLabel("art");
---

<Component translateLabel={translateLabel} />
```

Child components receive `translateLabel: TranslateLabel` where `TranslateLabel = (segment: string) => Record<string, string>` (locale → label for all locales in `SUPPORTED_LOCALES`) and render both as `data-locales`.

## Adding a new UI string

Add the key to every locale's map:

```ts
en: { myKey: "My String", ... },
pt: { myKey: "Minha String", ... },
```

Then use `t("myKey")` in React components (imported from `@i18n/labels`).
